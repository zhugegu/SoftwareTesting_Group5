## 引言

本报告基于黑盒与白盒测试方法，对 Python 标准库 `marshal` 模块的序列化稳定性进行系统性评估。`marshal` 用于生成 `.pyc` 文件，其输出应满足：相同输入（对象）始终产生完全相同的字节流。我们考察操作系统、Python 版本、浮点特殊值、递归结构、集合大小等因素对稳定性的影响。

## 测试执行与结果

### 5\.1 稳定性发现

关键缺陷：`float('nan')` 的序列化结果在同一 Python 版本、不同操作系统之间可能不同（因底层浮点库对 NaN 载荷位的处理不一致）。此外，同一进程内多次 marshal 同一个 NaN 对象是否总是得到相同字节流？实测偶现不同（取决于 NaN 的生成方式）。

### 5\.2 非预期行为列表

1. NaN 非确定性：`marshal.dumps(float('nan'))` 在多次调用中可能产生两个不同的字节序列（即使在同一进程中）。这是因为 `float('nan')` 每次可能生成不同的 NaN 位模式（如不同的静默 NaN 载荷）。建议：若依赖确定性，应避免封送 NaN。

2. 复数中的 NaN：同样问题扩展到复数实部/虚部。

3. 集合/字典中的 NaN：因为 NaN 不与自身相等，但作为键时会影响哈希，序列化后的字节流仍可能因 NaN 位模式而变化。

### 6\.1 完整性

- 覆盖了 `marshal` 文档列出的所有受支持类型。

### 6\.2 局限性

- 未测试版本间兼容性：`marshal` 本就不保证跨版本稳定，所以未作为缺陷报告。

- 未测试 `code` 对象的所有字段：`code` 对象包含大量元数据，生成所有可能状态较困难。

- 未测试并发场景：多线程同时 marshal 同一对象可能产生竞态？但 `marshal` 不保证线程安全，不属于“相同输入”范畴。

- 未测试内存映射文件：`marshal.loads` 的输入来源不影响稳定性，故省略。

- 模糊测试的随机性：虽重复运行多次，但不可能穷尽所有可能对象。

---

## 结论

`marshal` 模块在常规使用下（无 NaN 值）表现出良好的稳定性：相同输入在同一 Python 版本和同一平台上总是产生相同的字节流。跨平台稳定性也基本满足，唯一的显著例外是 NaN 浮点值的序列化结果不稳定，这源于 IEEE 754 允许多种 NaN 表示以及 Python 的 `float('nan')` 未规范位模式。

建议：如果用户需要绝对确定性的序列化（例如用于哈希缓存），应避免在对象中包含 NaN，或使用 `pickle` 的固定协议版本（但 pickle 也有类似问题）。`marshal` 的设计目标并非长期存储，因此上述非确定性是可接受的，但仍需在文档中明确警告。
