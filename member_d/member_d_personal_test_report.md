报告日期：2026-06-08
作者：成员 D
代码仓库：https://github.com/zhugegu/SoftwareTesting_Group5.git

## 摘要

本报告针对 Python 标准库 `marshal` 模块中成员 D 负责的测试范围进行总结。成员 D 的核心任务是验证 `marshal` 在复杂结构和异常输入下的稳定性与正确性，重点包括递归/循环引用结构、深层嵌套边界、不支持类型、共享引用、`marshal.dumps()` 的 version 参数，以及 `marshal.loads()` 面对非法字节流时的异常行为。

测试目标围绕课程项目提出的核心问题展开：相同输入是否总能产生字节级完全一致的 marshal 输出。对于可成功序列化的对象，本测试通过重复执行 `marshal.dumps()` 并比较 SHA-256 哈希值验证同进程确定性；对于不支持类型、过深嵌套结构和非法反序列化输入，本测试关注其是否能够稳定抛出一致异常，而不是出现崩溃、卡死或非确定性行为。

本测试套件首先在本地 Windows AMD64 + Python 3.13 环境下运行，通过后进一步使用 GitHub Actions 在 Ubuntu、Windows、macOS 三类操作系统，以及 Python 3.9、3.10、3.11、3.12、3.13 五个版本上运行，共覆盖 15 个执行环境。测试结果显示：成员 D 范围内未观察到同进程非确定性输出；循环引用、共享引用、深层嵌套边界、version 参数和非法 bytes 反序列化行为均表现稳定；记录到的异常均属于预期边界或不支持输入行为，不作为 marshal 缺陷处理。

## 1. 负责分工与需求提炼

成员 D 负责的测试范围为：递归/循环结构、异常输入、共享引用、协议版本参数和反序列化异常流。该部分主要对应项目需求中的 R5 类复杂结构与异常流测试，同时扩展了 `marshal.loads()` 的非法字节流测试，使测试范围同时覆盖序列化和反序列化。

| 需求 ID | 需求名称                 | 详细描述                                                                  | 成员 D 覆盖方式                                           |
| ----- | -------------------- | --------------------------------------------------------------------- | --------------------------------------------------- |
| R5.1  | 循环引用安全检测能力           | 当输入对象包含直接或间接循环引用时，marshal 应稳定处理，不应崩溃或卡死。                              | 测试自引用 list、自引用 dict、间接循环 list。                      |
| R5.2  | 嵌套极限边界稳定性            | 在嵌套深度边界附近，合法深度应稳定序列化，超过限制应稳定抛出异常。                                     | 测试深度 10、100、500、900、1000 的嵌套 list 和 tuple。          |
| R5.3  | 不支持类型异常一致性           | 对 generator、function、自定义对象等不支持类型，应稳定拒绝并抛出一致异常。                        | 测试 object、自定义类实例、普通函数、lambda、generator、file object。 |
| R5.4  | 共享引用输出确定性            | 同一对象在复杂结构中被多次引用时，序列化输出应稳定，反序列化后引用关系应正确。                               | 测试 `[shared, shared]` 和嵌套共享引用结构。                    |
| R5.5  | marshal version 参数行为 | 同一对象在不同 marshal format version 下输出可以不同，但同一 version 内应保持稳定。            | 测试 `version=0` 到当前 `marshal.version` 的重复序列化行为。      |
| R5.6  | 非法字节流反序列化异常流         | `marshal.loads()` 面对空 bytes、截断 bytes、随机 bytes 和尾随 extra bytes 时应表现稳定。 | 测试非法 bytes 稳定失败，以及合法数据后追加 extra bytes 的稳定读取行为。      |

## 2. 测试策略

### 2.1 使用的测试技术

| 测试技术      | 使用位置                               | 说明                                                     |
| --------- | ---------------------------------- | ------------------------------------------------------ |
| 等价类划分     | 循环结构、不支持类型、共享引用、非法 bytes           | 将复杂输入分成不同输入类别，分别选择代表性对象进行测试。                           |
| 边界值分析     | 深层嵌套 list/tuple、marshal version 参数 | 对深度 10、100、500、900、1000 进行边界测试，并覆盖 version 参数 0 到当前版本。 |
| 异常流测试     | 不支持类型、过深嵌套、非法反序列化字节流               | 验证异常类型和异常信息是否稳定，不将预期拒绝误判为缺陷。                           |
| 重复执行稳定性测试 | 所有可序列化 case                        | 每个输入重复执行 100 次 `marshal.dumps()`，比较 SHA-256 哈希唯一数量。    |
| 正确性测试     | 循环引用、共享引用、合法 loads 场景              | 不只比较 hash，还使用 `marshal.loads()` 检查反序列化后结构和引用关系是否正确。    |
| 白盒导向测试    | 递归结构、引用缓存、类型分发、version 参数          | 根据 marshal 对递归对象、共享引用、类型支持和版本参数的内部处理逻辑设计输入。            |

### 2.2 为什么不使用某些技术

本分工没有采用大规模随机 fuzzing 作为主方法。原因是成员 D 的目标并不是随机发现所有潜在崩溃，而是围绕明确需求验证递归结构、异常输入和引用行为的稳定性。随机 fuzzing 可能大量生成容器、整数、浮点数等对象，容易与成员 A、B、C 的分工重复。
本分工也没有采用变异测试，因为 marshal 输出是二进制字节流，变异后的正确性判定成本较高，而本任务的主要关注点是确定性和稳定异常行为。形式化验证也未采用，因为 marshal 是 Python 内部格式，并不是跨版本长期稳定协议，缺少适合形式化证明的公开稳定规范。

### 6.3 主要稳定性发现

第一，循环引用结构在测试环境中表现稳定。自引用 list、自引用 dict 和间接循环 list 均能够稳定序列化，并且反序列化后引用关系正确保留。这说明在当前测试覆盖范围内，marshal 对递归容器的引用处理具有确定性。

第二，深层嵌套结构表现出明确边界。深度 10、100、500、900 的嵌套 list 和 tuple 均能稳定序列化；深度 1000 的 list 和 tuple 稳定抛出 `ValueError: object too deeply nested to marshal`。该行为属于预期边界拒绝，不视为缺陷。

第三，不支持类型的异常行为一致。`object()`、自定义类实例、普通函数、lambda、generator 和 file object 均稳定抛出 `ValueError: unmarshallable object`。这说明 marshal 能够稳定拒绝不在支持范围内的对象。

第四，共享引用结构表现稳定。对于 `[shared, shared]` 和嵌套共享引用结构，重复序列化哈希稳定，反序列化后共享引用关系也能正确保留。该结果不仅验证了稳定性，也验证了结构正确性。

第五，marshal version 参数行为符合预期。不同 version 参数可能产生不同字节流，但同一 version 参数下重复序列化结果保持稳定。因此，version 差异应记录为格式差异，而不是非确定性缺陷。

第六，非法字节流反序列化行为稳定。空 bytes、单字节非法输入、随机 bytes 和截断 marshal 数据均稳定失败；合法 marshal 数据后追加 extra bytes 能够被稳定读取，反序列化结果正确。这说明 `marshal.loads()` 在本测试范围内具有稳定异常流行为。

## 8. 测试套件完整性与局限性

### 8.1 完整性

成员 D 测试套件覆盖了复杂结构与异常输入相关的主要风险点，包括直接循环引用、间接循环引用、深层嵌套边界、不支持类型、共享引用、version 参数和非法 bytes 反序列化。测试不只检查 `marshal.dumps()` 是否产生稳定字节流，也检查了部分 `marshal.loads()` 的结构正确性，例如自引用关系和共享引用关系是否被保留。

从测试方法角度看，本测试覆盖了等价类划分、边界值分析、异常流测试、重复执行稳定性测试和白盒导向测试。所有 case 均记录到 JSON 原始结果，并通过 Markdown summary 和跨平台 summary 进行汇总，便于复核。

### 8.2 局限性

第一，测试对象虽然覆盖了典型复杂结构，但没有穷尽所有可能的递归组合。例如，多层 dict/list/set 混合循环、包含复杂 key 的递归结构、以及更大规模的共享引用图仍未完全覆盖。

第二，深层嵌套测试使用了固定深度 10、100、500、900、1000。虽然这些值可以体现边界值分析，但没有进一步二分搜索 marshal 的精确最大可处理深度。

第三，非法字节流测试使用了少量确定性输入，没有进行大规模随机 fuzzing。因此，该部分只能说明当前选定非法输入下异常行为稳定，不能证明所有畸形 bytes 都安全稳定。

第四，测试运行环境依赖 GitHub-hosted runners，覆盖了常见 Windows、Linux、macOS 环境，但不能代表所有真实硬件、操作系统补丁版本、32 位架构或非 CPython 解释器。

第五，测试重点是稳定性和正确性，不是安全性。`marshal` 本身不适合处理不可信来源数据，本测试没有尝试构造恶意 payload 或进行安全攻击测试。

---

## 9. 结论

成员 D 测试套件完成了递归结构、异常输入、共享引用、marshal version 参数和非法字节流反序列化行为的稳定性测试。测试结果表明，在当前 15 个 GitHub Actions 环境中，成员 D 范围内未观察到同进程非确定性输出。循环引用、共享引用、合法深层嵌套和 version 参数测试均表现出稳定行为；过深嵌套和不支持类型能够稳定抛出预期异常；非法 bytes 反序列化也表现出一致的失败或成功行为。

因此，在成员 D 测试范围内，可以认为 `marshal` 对复杂结构和异常输入具有良好的确定性和稳定异常处理能力。需要注意的是，该结论仅适用于当前测试输入、CPython 版本和 GitHub Actions 覆盖环境，不能扩展为对所有 Python 实现、所有硬件架构和所有可能对象图的完全证明。
